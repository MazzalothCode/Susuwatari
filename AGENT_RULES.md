你现在参与的是一个长期项目。

从现在开始，你必须持续维护：

PROJECT_CONTEXT.md

这是整个项目的长期记忆文档。

规则如下：

# 文档目标

PROJECT_CONTEXT.md 必须让：

* 新开的 Agent
* 完全没有上下文的 Codex
* context 被压缩后的 session

仍然能够快速理解整个项目。

文档必须长期可读、结构化、可恢复项目状态。

---

# 更新规则（极其重要）

禁止：

* 覆盖整个文件
* 重写旧内容
* 压缩历史记录
* 删除旧结论

必须：

* 使用“追加更新”的方式
* 保留历史时间线
* 每次新增内容都写明时间与阶段
* 保留被证伪的假设

永远不要把旧记录直接替换掉。

---

# 每次工作后必须更新

每完成以下任一行为后：

* 修改代码
* 验证假设
* 调试问题
* 确认架构
* 排除问题
* 发现新线索

都必须更新 PROJECT_CONTEXT.md。

即使结论失败，也必须记录。

---

# PROJECT_CONTEXT.md 必须包含以下结构

## 1. 项目概览

包括：

* 项目用途
* 技术栈
* 核心模块
* 当前主要功能
* 当前已知问题

---

## 2. 当前架构

记录：

* 状态流
* 事件流
* 模块关系
* Rive / React / Overlay / Pointer 等交互链路
* 哪些组件负责什么

必须避免未来 Agent 重新推测架构。

---

## 3. 已确认事实（非常重要）

必须记录：

* 已确认行为
* 已确认运行机制
* 已确认限制
* 哪些地方“肯定不是问题”

例如：

* listener 在 component 内工作正常
* overlay 会收到全局 pointerup
* Brush 状态会被多来源覆盖

这些事实不能丢失。

---

## 4. 已失败假设（极其重要）

必须长期保留：

* 哪些假设已经被证伪
* 为什么失败
* 如何验证失败

避免未来 Agent 重复调试。

---

## 5. 调试时间线

按时间追加：

### [时间]

* 做了什么
* 改了哪些文件
* 为什么改
* 结果如何
* 是否成功
* 下一步是什么

必须像实验日志一样记录。

---

## 6. 当前工作状态

必须始终维护：

* 当前正在调查的问题
* 当前唯一假设
* 下一步计划
* 当前禁止修改的区域

---

# 调试规则（非常重要）

禁止：

* 在没有观测的情况下重构架构
* 用“统一状态源”替代问题定位
* 在没有证据时修改多个系统

必须：

* 先插桩
* 先观测
* 先证明
* 再修复

---

# 工作方式

每次开始工作前：

1. 先阅读 PROJECT_CONTEXT.md
2. 总结当前状态
3. 确认当前唯一目标
4. 不允许重新探索已经确认的内容

---

# 输出规则

更新 PROJECT_CONTEXT.md 时：

* 使用 append 追加
* 不要覆盖旧内容
* 保持 markdown 结构
* 保留历史记录
* 保留失败实验
* 保留时间线

PROJECT_CONTEXT.md 是整个项目的长期记忆核心。

每当 PROJECT_CONTEXT.md 超过一定长度：

禁止压缩历史。

而是：

新增：

ARCHIVE_001.md
ARCHIVE_002.md

PROJECT_CONTEXT.md 只保留：
- 当前架构
- 当前状态
- 最近调试
- 索引

旧记录移入 archive。

长期维护KNOWN_FACTS.md
里面只放：
100% 确认的事实
